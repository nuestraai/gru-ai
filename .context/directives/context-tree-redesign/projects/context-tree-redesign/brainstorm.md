## Sarah Chen (CTO) — Data Model First

### Core Thesis
The problem is not "too many folders" — it's that the folder structure doesn't match the entity model. We have 6 entity types (goals, directives, work-items, intelligence, lessons, reports) spread across 10+ directories with no consistent pattern. The fix: one directory per entity type, one schema per entity type, one file per entity instance. Delete everything that doesn't map to a real entity.

### Proposed Structure

```
.context/
├── vision.md                    # System vision (framework-level)
├── preferences.md               # CEO preferences (if applicable)
├── goals/
│   └── {goal-id}/
│       ├── goal.json            # Goal metadata + features array (SINGLE SOURCE OF TRUTH)
│       └── backlog.json         # Prioritized ideas with triggers
├── directives/
│   ├── {directive-id}.json      # Structured metadata (status, goal_ids, plan, dod, etc.)
│   ├── {directive-id}.md        # Human-readable content (problem, what we want, constraints)
│   └── ...
├── reports/
│   └── {directive-id}-{date}.md # Execution reports (markdown content)
├── intelligence/
│   └── {agent}-{date}.json      # Scout outputs (structured findings)
├── lessons/
│   ├── _index.md                # Lessons index with topic table
│   ├── orchestration.md
│   ├── agent-behavior.md
│   ├── state-management.md
│   ├── review-quality.md
│   └── skill-design.md
└── _state/                      # Generated by indexer (gitignored or committed)
    ├── goals.json
    ├── features.json
    ├── backlogs.json
    ├── directives.json
    ├── references.json
    └── index.json
```

### Key Design Decisions

**1. Flatten directives into one directory.** Kill inbox/ and done/. Directive status lives in the .json file (`status: "pending" | "executing" | "done"`). The filesystem isn't the state machine — the JSON is. Agents filter by status field, not by directory.

**2. Absorb artifacts INTO directive.json.** Morgan's plan, Sarah's review findings, build outputs — these are all phases of a directive. Store them as structured fields in `directive.json` rather than as separate markdown files in a nested artifacts/ tree. The directive.json becomes the complete lifecycle record:
```json
{
  "id": "context-tree-redesign",
  "status": "executing",
  "plan": { "initiatives": [...], "challenges": [...] },
  "phases": {
    "challenge": { "sarah": {...}, "morgan": {...} },
    "audit": { "findings": [...] },
    "build": { "completed_tasks": [...] },
    "review": { "outcome": "pass", "findings": [...] }
  },
  "dod": { "criteria": [...] },
  "report": "context-tree-redesign-2026-03-03"
}
```

**3. Kill discussions/ — they're just directives or lessons.** The 4 existing discussions are either context that should be a lesson (and referenced from lessons/) or the genesis of a directive (and should be mentioned in the directive's .md content). Standalone discussion files are write-once graveyards.

**4. Kill artifacts/ entirely.** Nothing references artifacts after a directive completes. The build.md and review.md files are intermediate outputs that belong in directive.json phases. The brainstorm.md and plan-for-approval.md are planning outputs that also belong in directive.json (or as the .md content itself).

**5. Rename state/ to _state/ with underscore prefix** to signal it's generated/computed, not authored.

**6. Keep goal.md and okrs.md as optional narrative files.** goal.json is the source of truth for structured data. goal.md provides the narrative "why" that agents read for context. okrs.md defines measurable objectives. These serve different purposes and both earn their existence.

**7. Kill backlog.md.** backlog.json replaced it. The .md version is pure legacy.

### Entity Schemas

| Entity | File | Schema Summary |
|--------|------|----------------|
| Goal | `goals/{id}/goal.json` | id, title, state, category, description, features[], last_activity |
| Feature | (inside goal.json) | id, title, status, source_directive, source_backlog, dod, report, refs |
| Backlog Item | `goals/{id}/backlog.json` | id, title, status, priority, trigger, source_directive, promoted_to_feature |
| Directive | `directives/{id}.json` | id, title, status, weight, goal_ids, plan, phases, dod, produced_features, report |
| Report | `reports/{id}.md` | Markdown content (human-readable execution summary) |
| Intelligence | `intelligence/{agent}-{date}.json` | agent, date, findings[], proposals[] |
| Lesson | `lessons/{topic}.md` | Markdown content (categorized by topic) |

### Tradeoffs
- **directive.json becomes large** (plan + phases + dod) vs many small files. Worth it: one entity = one file. The agent reads one file to understand the full directive lifecycle.
- **No feature directories.** Features are metadata in goal.json, not filesystem entities with their own directories. This means no spec.md or tasks.json per feature. Tasks become fields in the directive's plan. This is a simplification trade — lose per-feature filesystem artifacts, gain schema consistency. For the agent-conductor repo (where most features are JSON-only anyway), this is fine.
- **Intelligence loses latest/ directory.** Each scout run produces a dated file. "Latest" is derived by the indexer (sort by date, take first). No need for a latest/ symlink/copy mechanism.

### What to Avoid
Do NOT create a database or single monolithic state file. The file-per-entity pattern works with git (meaningful diffs, no merge conflicts). Do NOT keep artifacts/ as a separate directory — it's where context goes to die. Do NOT keep inbox/done split — directory-as-status is a poor abstraction when the status is a field in a JSON file.

### Confidence: High

## Marcus Rivera (CPO) — User Journey First

### Core Thesis
Organize files by how people FIND them, not by what they ARE. The CEO's journey is: "What's happening?" → goals dashboard → "What directive is running?" → directive details → "How did it go?" → report. An agent's journey is: "What should I work on?" → read directive → "What context do I need?" → read goal + lessons → "Where do I write results?" → update directive. The tree should follow these navigation paths.

### Proposed Structure

```
.context/
├── vision.md
├── goals/
│   └── {goal-id}/
│       ├── goal.json            # Structured goal data (features, status)
│       ├── about.md             # Narrative context (optional, replaces goal.md + okrs.md combined)
│       └── backlog.json         # Ideas and deferred items
├── work/
│   ├── {directive-id}/
│   │   ├── directive.json       # The work item: status, plan, DOD, phases, outcomes
│   │   ├── brief.md             # Human-readable problem statement + requirements
│   │   └── report.md            # Execution report (written at completion)
│   └── ...
├── intel/
│   └── {date}/
│       ├── sarah.json
│       ├── marcus.json
│       ├── morgan.json
│       └── priya.json
├── knowledge/
│   ├── _index.md
│   ├── orchestration.md
│   ├── agent-behavior.md
│   ├── state-management.md
│   ├── review-quality.md
│   └── skill-design.md
└── _state/                      # Indexer output
    └── *.json
```

### Key Design Decisions

**1. Introduce `work/` as the single work container.** Every directive gets its own directory under `work/`. Inside that directory lives everything about that work item: the brief (.md), the structured data (.json), and the report (.md). No more scattering a directive's lifecycle across inbox/, done/, artifacts/, and reports/.

This is inspired by how Linear works: an issue is one entity with attachments, comments, and state — all co-located. You don't have separate "inbox issues" and "done issues" directories.

**2. directive.json becomes the "work item ticket."** It contains:
- Status lifecycle (pending → planning → executing → reviewing → done)
- Morgan's plan (initiatives array — persisted, not ephemeral)
- Sarah's challenge response
- DOD criteria with verification status
- Phase outputs (audit findings, build summary, review outcome)
- Links to goal_ids, produced_features, backlog_sources

**3. Combine goal.md + okrs.md into one `about.md`.** Two separate narrative files for a goal is excessive. One file with sections for "About," "Objectives," and "Key Results" serves the same purpose. goal.json remains the structured source of truth.

**4. Rename lessons/ to `knowledge/`.** "Lessons" implies past mistakes. This directory also contains patterns, conventions, and how-things-work documentation. "Knowledge" is more accurate and matches the CEO's read pattern: "I need to understand how X works" → look in knowledge/.

**5. Rename intelligence/ to `intel/` and organize by date.** Scout runs happen on a schedule (Monday). Group by date so the CEO can see "what did the scouts find this week?" Each date directory has 4 JSON files. The indexer derives "latest" from the most recent date directory.

**6. Kill: artifacts/, discussions/, checkpoints/, *.log files.**
- Artifacts → absorbed into work/{directive}/directive.json phases
- Discussions → either become knowledge/ entries or are referenced in directive briefs
- Checkpoints → stored in directive.json as a `checkpoint` field
- *.log files → empty, never used

### Entity Schemas

| Entity | Location | Key Fields |
|--------|----------|------------|
| Goal | `goals/{id}/goal.json` | id, title, state, category, description, features[], last_activity |
| Feature | Inside goal.json features[] | id, title, status, source_directive, dod, report |
| Backlog Item | `goals/{id}/backlog.json` | id, title, status, priority, trigger, source_directive |
| Work Item | `work/{id}/directive.json` | id, title, status, weight, goal_ids, plan, phases, dod, report_summary |
| Report | `work/{id}/report.md` | Markdown execution report |
| Brief | `work/{id}/brief.md` | Markdown problem statement + requirements |
| Intel | `intel/{date}/{agent}.json` | agent, findings[], proposals[], date |
| Knowledge | `knowledge/{topic}.md` | Markdown topical lessons |

### Tradeoffs
- **work/ directories per directive add filesystem overhead** (one dir per directive vs one file). Worth it: co-locating brief + data + report makes the directive self-contained. An agent reads one directory to get everything.
- **Renaming familiar paths** (lessons → knowledge, intelligence → intel) creates a learning curve. Worth it: the new names more accurately describe what's inside.
- **Losing separate reports/ directory** means reports aren't browsable as a flat list. The indexer compensates — _state/ provides the flat aggregated view.

### What to Avoid
Do NOT keep the inbox/done directory split. Status belongs in data, not in the filesystem. Do NOT maintain parallel formats (backlog.md + backlog.json). Do NOT create deeply nested structures (artifacts/directive/initiative/phase.md is 4 levels deep for a single file).

## Morgan Park (COO) — Operations & Pipeline First

### Core Thesis
The structure should mirror the directive pipeline, because that's the primary write workflow. Today the pipeline scatters data across 4+ directories. Fix: each directive is a self-contained folder with all its lifecycle data. Goals are the other axis — they're the persistent context that outlives any single directive. Everything else (lessons, intelligence) is supporting context.

### Proposed Structure

```
.context/
├── vision.md
├── goals/
│   └── {goal-id}/
│       ├── goal.json            # Goal metadata + features
│       └── backlog.json         # Prioritized work ideas
├── directives/
│   ├── pending/                 # Not yet picked up
│   │   └── {id}.md             # Just the brief — no JSON until triage
│   ├── active/                  # Currently executing (max 1-2)
│   │   └── {id}/
│   │       ├── brief.md        # Original directive text
│   │       ├── directive.json  # Full lifecycle: plan, audit, build, review, DOD
│   │       └── checkpoint.json # Resume state (if context exhaustion occurs)
│   └── done/                   # Completed
│       └── {id}/
│           ├── brief.md
│           ├── directive.json  # Final state with all phases complete
│           └── report.md       # Execution report
├── intel/
│   └── latest/                 # Overwritten each scout cycle
│       ├── sarah.json
│       ├── marcus.json
│       ├── morgan.json
│       └── priya.json
├── lessons/
│   ├── _index.md
│   └── {topic}.md
└── _state/
    └── *.json
```

### Key Design Decisions

**1. Directives get status-based subdirectories (pending/active/done).** I know Sarah will argue against this (she prefers status-in-JSON). Here's why I disagree for THIS system: the directive pipeline is a physical process. `pending/` is a queue. `active/` is the workbench (max 1-2 items — this is a constraint, not a feature). `done/` is the archive. Agents list `active/` to know what to work on. They don't need to parse JSON and filter.

But here's my concession: pending/ directives are just .md files (no JSON). They get triaged into active/ where the directive.json is created during Step 0. This matches the actual workflow: the CEO writes a problem statement, the system structures it during triage.

**2. directive.json is the unified work item.** It replaces: tasks.json (initiatives become tasks inside directive.json), Morgan's ephemeral plan (persisted as `plan` field), artifacts (phase outputs stored inline), DOD (structured criteria with verification), checkpoint (inline or companion file).

Schema:
```json
{
  "id": "context-tree-redesign",
  "title": "Context Tree Redesign — Clean Slate",
  "status": "active",
  "weight": "strategic",
  "created": "2026-03-03",
  "goal_ids": ["agent-conductor"],
  "plan": {
    "challenges": [...],
    "initiatives": [
      {
        "id": "init-1",
        "title": "Design new structure",
        "status": "done",
        "cast": { "auditor": "sarah", "engineer": "unnamed", "reviewer": "sarah" },
        "phases": ["build", "review"],
        "dod": [
          { "text": "New .context/ tree documented", "met": true, "verified_by": "sarah" }
        ],
        "audit": { "findings": [...] },
        "build": { "summary": "...", "files_changed": [...] },
        "review": { "outcome": "pass", "findings": [...] }
      }
    ]
  },
  "produced_features": [],
  "backlog_sources": [],
  "report": "context-tree-redesign-2026-03-03"
}
```

**3. Kill standalone tasks.json entirely.** Tasks are initiatives inside directive.json. There is no separate task file. When a directive has 3 initiatives, each initiative IS a task with its own DOD, its own phase outputs, its own status. This kills the dual-system problem permanently.

For goal-level work (backlog items being worked without a directive), the backlog item gets `status: "in-progress"` — it doesn't spawn a parallel task system.

**4. Keep intel/latest/ as overwrite-on-each-scout-run.** Simplest possible intelligence storage. The indexer reads latest/. Historical intel is in git history. No date-based directories, no archiving. Scout runs weekly — 4 files overwritten each time.

**5. Kill: artifacts/, discussions/, reports/ (as standalone dir), *.log, backlog.md, scenarios.md.**
- reports/ absorbed into done/{directive}/report.md
- artifacts/ absorbed into directive.json phases
- discussions/ → migrate to lessons/ or directive briefs
- scenarios.md → migrate to lessons/ or delete if outdated
- *.log → delete (empty, never used)
- backlog.md → delete (replaced by backlog.json long ago)

### Entity Schemas

| Entity | Location | Write Moment |
|--------|----------|-------------|
| Goal | `goals/{id}/goal.json` | Updated at directive completion (features array) |
| Backlog | `goals/{id}/backlog.json` | Updated by /scout proposals and directive outcomes |
| Directive (pending) | `directives/pending/{id}.md` | CEO creates |
| Directive (active) | `directives/active/{id}/directive.json` | Triage → plan → build → review (progressive enrichment) |
| Directive (done) | `directives/done/{id}/directive.json` | Archived at completion with report.md |
| Intel | `intel/latest/{agent}.json` | Overwritten each /scout run |
| Lessons | `lessons/{topic}.md` | Updated at directive Step 6 |

### Tradeoffs
- **Directory-based status (pending/active/done) vs flat + status field.** I choose directories because the operational workflow benefits: `ls directives/active/` instantly shows what's in flight. The downside: `mv` operations during transitions. Acceptable — there are max 2 active directives at once.
- **No historical intel storage** (just latest/). Losing scout history. Acceptable — git history preserves it, and nobody has ever queried historical intel.
- **directive.json gets large** for complex directives (plan + 5 initiatives + phase outputs). Acceptable — one big file is better than 20 small scattered files.

### What to Avoid
Do NOT create a flat directives/ directory with 60+ .json files. Scrolling through 60 files to find the 1-2 active ones is terrible operations UX. Status-based subdirs solve this. Do NOT keep a separate artifacts/ directory — it's the single biggest contributor to the "over-engineered yet messy" problem. Do NOT keep parallel work-item systems (tasks.json + initiatives + backlog items all tracking progress differently).

## Key Design Tensions

### 1. Flat vs Directory-Based Status
The fundamental question: should directive status be encoded in the directory path or in a JSON field?

**For flat (Sarah):** Simpler filesystem, status is data not structure, no `mv` operations on transitions, the indexer doesn't care about directory location.

**For subdirs (Morgan):** Operational clarity (`ls active/` is instant), physical constraint on active work (you SEE only 1-2 items), matches the queue/workbench/archive mental model.

**Middle ground (Marcus):** Flat but with per-directive directories (for co-location), status in JSON.

### 2. Co-location vs Separation-by-Type
Should a directive's brief, data, and report live together (Marcus, Morgan) or in separate entity-type directories (Sarah)?

**For co-location:** Self-contained work items, an agent reads one directory for everything.

**For separation:** One schema per directory, cleaner for the indexer, no mixed file types.

### 3. How Much Goes in directive.json?
All three agree it should absorb plan + DOD + phase outputs. But how large should it get?

**Minimal (Sarah):** Structured metadata only. Phase outputs are summaries, not full build logs.

**Maximal (Morgan):** Everything including full audit findings, build file lists, review comments.

**Middle (Marcus):** Structured lifecycle data + summaries. Full content stays in companion .md files.

## Clarifying Questions for the CEO

**1. Flat directives/ or status-based subdirectories?**
Sarah proposes a single flat `directives/` directory with status in the JSON. Morgan proposes `directives/pending/active/done/` subdirectories to match the queue/workbench/archive workflow. Both work — the question is whether you prefer `ls directives/ | grep active` (data-driven) or `ls directives/active/` (filesystem-driven). With ~60 directives and growing, the flat list gets long. But status-based dirs require `mv` operations during transitions. Which operational model do you prefer?

**2. Should reports live inside directive directories or in a separate reports/ directory?**
Marcus and Morgan propose co-locating `report.md` inside the directive's directory (everything about one work item in one place). Sarah proposes keeping `reports/` as a standalone directory (reports are their own entity type, and sometimes the CEO wants to browse all reports chronologically). The dashboard already aggregates reports — does the CEO ever browse the reports/ directory directly, or only through the dashboard?

**3. How much should directive.json absorb?**
All three agree that Morgan's plan, DOD, and phase metadata should live in directive.json. But should it also store full audit findings, review comments, and build outputs? A large directive.json (5-10KB) is one file to read. A minimal directive.json (~2KB) plus separate phase files means more files but smaller reads. For AI agents consuming these files, which matters more: fewer files or smaller files?
